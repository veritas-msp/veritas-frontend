Vision globale
L’objectif n’est pas une « injection » dans un silo à part, mais un mode opératoire logistique branché sur l’inventaire Veritas existant — surtout les familles de matériel (v_b_equipment_family_definitions + v_b_clients_m_custom_equipment), déjà génériques et configurables par champ.

La logistique ne devrait pas repasser par la fiche entreprise ni le modal d’ajout équipement actuel (trop lourd pour enchaîner 20 scans). Il faut une surface dédiée, scan-first, qui écrit au même endroit que le reste de l’app.

Fiche entreprise
Pourquoi pas un module d’injection isolé : vous dupliqueriez séries, factures, liens client, et il faudrait resynchroniser. La logistique doit être un accélérateur de saisie, pas un second inventaire.

Parcours utilisateur (perte de temps minimale)
Cible : 3 gestes répétés, le reste mémorisé dans la session.

Étape 0 — Une fois par session (30 s)
Entreprise — recherche rapide + raccourcis « clients récents »
Famille — chips des familles actives (Switch, PC, Onduleur, ou familles custom admin)
(Optionnel) Contexte sortie — n° BL, date, opérateur (prérempli)
Étape 1 — Boucle scan (cœur du flux)
Champ scan toujours focus (douchette USB = clavier)
Chaque scan → ligne dans une liste (pas de modal)
Feedback immédiat : son / flash vert ou orange si doublon
Auto-remplissage selon le code :
N° de série seul → champ numero_serie
EAN/GTIN → lookup catalogue interne (si vous l’ajoutez) → marque/modèle
QR métier (JSON) → { family, serial, model, ... }
Étape 2 — Validation groupée
Récap : X lignes pour Entreprise Y
Corriger inline une ligne si besoin
Valider tout → N inserts en une transaction
Option : créer ticket « Livraison matériel » ou note sur le contrat (phase 2)
Sur mobile : bouton caméra en complément de la douchette, pas en remplacement.

Généricité (toutes entreprises, toutes familles)
Vous avez déjà la bonne base admin :

Familles activables + champs configurables (texte, date, booléen…)
Champs par défaut utiles logistique : numero_serie, numero_facture, date_facturation, marque, modele
Principe : le module logistique ne connaît pas « un switch Stormshield », il connaît :

une famille (family_key)
ses champs (schéma dynamique)
une règle de mapping scan → champs (config admin légère)

Points d’attention
Legacy vs custom — Aujourd’hui coexistent équipements typés (Serveurs, NAS…) et familles custom. Pour rester générique : viser les familles custom en priorité ; pont vers legacy seulement pour familles « mappées » (ex. Stockage → NAS).
Doublons — Règle claire : même S/N chez le même client = alerte, pas blocage silencieux.
Audit — La logistique a besoin de traçabilité (qui a scanné quoi, quand) ; même un log simple évite les litiges.
Hors-ligne — Pas au MVP ; session locale + sync si le réseau coupe peut venir plus tard.
Recommandation synthétique
Oui à un module navigation dédié « Logistique / Sortie de stock », optimisé scan + validation groupée.
Non à un silo d’injection séparé : brancher sur familles + custom_equipment.
Généricité via le modèle familles/champs déjà en admin, plus règles de mapping scan configurables.
MVP : entreprise + famille + liste de scans + commit batch ; douchette d’abord, caméra ensuite.
Si tu veux passer à l’implémentation, je proposerais de commencer par le MVP (page + API batch + droits profil) avant toute couche stock interne. On peut détailler le périmètre MVP (champs obligatoires, formats de codes, lien ticket/contrat) si tu as des contraintes métier côté logistique.
